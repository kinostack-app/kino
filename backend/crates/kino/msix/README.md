# MSIX packaging

This directory holds the Microsoft Store / sideloadable MSIX
packaging for kino. Mirrors the `debian/` and `rpm/` siblings: source
of truth for one Windows distribution channel, version-controlled
alongside the code.

## Layout

```
msix/
├── appxmanifest.xml   # Package manifest (capabilities, service, identity)
├── Assets/            # Tile PNGs (regenerated by build_msix_tiles.py)
└── README.md          # This file
```

## How it ships

The `build-msix` job in `.github/workflows/release.yml` runs on
`windows-latest` after the `x86_64-pc-windows-msvc` matrix build:

1. Downloads the cargo-dist Windows artefact (`kino.exe` + sidecars).
2. Stages a flat layout: `appxmanifest.xml`, `Assets/`, `kino.exe`.
3. Templates the placeholder Identity values + the four-part
   version string into the manifest.
4. Runs `makeappx pack` (Windows SDK; preinstalled on the runner) to
   produce `kino-<version>-x64.msix`.
5. Uploads it to the GitHub Release.

The Microsoft Store **re-signs the MSIX** on submission, so CI does
not sign it. Sideload testing on a dev machine does require a sig —
see "Sideload testing" below.

The `publish-msstore` job in `.github/workflows/channels.yml` then
either submits via the `msstore` CLI (if the four MSSTORE_* secrets
plus the `MSSTORE_PRODUCT_ID` variable are configured) or soft-skips
with a notice — same pattern as the winget / AUR jobs.

The first submission for a new product **must be done manually
through Partner Center** (Microsoft requirement); the API can only
manage submissions on a product that already has at least one
manually-published submission. So v0.1.0 is a manual upload; v0.2.0+
is the automated path.

## Regenerating the tile assets

The five PNGs under `Assets/` are downscaled from
`packaging/icons/linux/kino-512.png` via ImageMagick. From the
repo root:

```sh
python3 packaging/icons/build_msix_tiles.py
```

Output is deterministic (`-strip` flag drops metadata so git
diffs stay clean). Re-run after editing the master 512px source.

## Bumping the manifest

Two values are placeholders, replaced by CI before `makeappx pack`:

| Placeholder | Replaced with | Source of truth |
|---|---|---|
| `PLACEHOLDER-PUBLISHER` | The Identity Name + Publisher CN string Microsoft assigns to the kinostack-app account | Partner Center → Product identity (visible after Microsoft completes identity verification + first product creation) |
| `0.0.0.0` | The release version, four-part SemVer (`X.Y.Z.0`) | Git tag (cargo-dist already passes the three-part version through; CI appends `.0`) |

Both replacements live in the `build-msix` job — see
`.github/workflows/release.yml`. To bump permanently (e.g. you've
been issued the real publisher ID), update both this file and the
manifest's `<Identity>` block in the same commit so local `makeappx`
runs work without the CI templating.

## Sideload testing

For testing the MSIX on a dev Windows host (i.e. without going
through the Store):

1. Generate a self-signed cert (one-time):
   ```powershell
   winget install microsoft.winappcli   # if not already installed
   winapp cert generate --if-exists skip
   winapp cert install .\devcert.pfx     # admin shell — adds to Trusted Root
   ```

2. Pack with the dev cert (use the same `makeappx pack` invocation
   as CI, then sign):
   ```powershell
   makeappx pack /d msix-stage /p kino-dev.msix /o
   signtool sign /fd SHA256 /a /f devcert.pfx /p <password> kino-dev.msix
   ```

3. Install:
   ```powershell
   Add-AppxPackage .\kino-dev.msix
   ```

4. Inspect:
   ```powershell
   Get-AppxPackage *Kino*           # confirm the package registered
   Get-Service kino                 # confirm the service registered via the manifest
   sc query kino
   ```

Uninstall: `Get-AppxPackage *Kino* | Remove-AppxPackage`.

## Known limitations (v0.1.0)

- **No tray autostart on MSIX.** `<windows.startupTask>` doesn't
  support custom command-line args, so a launcher binary is needed
  to invoke `kino tray`. Out of scope for first submission; tracked
  as a follow-up. MSIX users start the tray manually with
  `kino tray` or pin a Start Menu shortcut.
- **x64 only.** `Cargo.toml [workspace.metadata.dist].targets`
  doesn't include `aarch64-pc-windows-msvc`. When that's added,
  this package becomes an MSIX bundle (one MSIX per arch + a
  `.msixbundle` umbrella).
- **No auto-update.** Subsystem 27 (in-app self-update) detects the
  MSIX install path and disables itself; Store-managed updates take
  over. See `docs/roadmap/27-auto-update.md`.
