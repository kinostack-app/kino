# API subsystem

REST API (JSON over HTTP) — the sole interface between kino's backend and all clients. The Web UI, Cast receiver, and any external integrations consume this API exclusively.

## Design principles

- **Always paginate** — every list endpoint returns cursor-based paginated results. No dumping entire collections; libraries scale into the thousands.
- **Nest owned resources** — episodes live under `/shows/:id/seasons/:num/episodes`, not as flat paths with filter params. Clear ownership.
- **Singletons are singletons** — `GET /config`, `PUT /config`. No fake `/{id}` on single-row resources.
- **Named tasks, not RPC** — `POST /tasks/wanted-search/run`, not `POST /command {"name": "WantedSearch"}`. Discoverable without docs.
- **Consistent shapes** — same pagination envelope, same error shape, everywhere.
- **Lean defaults** — return the core resource. Use `?expand=` for related data.
- **snake_case JSON** — idiomatic for Rust/serde, matches TMDB response format.
- **OpenAPI spec** — the API is defined as an OpenAPI 3.x spec. The frontend TypeScript client is auto-generated from it via hey-api, keeping client and server in sync.

## Authentication

Single-user, but with three credentialing paths layered on the same
API-key root of trust:

- **Master API key** — generated on first boot (random UUID stored in
  the config table). Accepted via `Authorization: Bearer {key}` or
  `X-Api-Key: {key}` header. Used by the SPA's paste-the-key screen
  on first load, by the QR pairing flow, and by external integrations.
- **Session cookie** — `kino-session`. Issued by `POST /api/v1/sessions`
  in exchange for the master key, OR auto-issued by `GET /api/v1/bootstrap`
  for same-host browsers (no proxy hops; loopback-bound). Per-device
  revocation lives at Settings → Devices. The everyday browser path.
- **Signed media URLs** — short-lived HMAC-signed `?sig=...` query
  strings minted by `POST /api/v1/sessions/sign-url` for cross-origin
  media playback (Cast receiver, native clients) where cookies don't
  carry. Sig commits to method + path + expiry; constant-time
  verified.

A handful of bootstrap-time endpoints are public (no credential
required): `/api/v1/status`, `/api/v1/bootstrap`, `/api/v1/sessions`
(POST — the body IS the auth), and a couple of QR redemption
endpoints. Everything else requires one of the three above.

## Pagination

Cursor-based pagination on all list endpoints. Designed for infinite scroll — no page numbers, no "page drift" when items are added or removed.

```
GET /api/v1/movies?limit=25&sort=title&order=asc

{
  "results": [...],
  "next_cursor": "eyJpZCI6NDJ9",
  "has_more": true
}
```

Next page:
```
GET /api/v1/movies?limit=25&sort=title&order=asc&cursor=eyJpZCI6NDJ9
```

- `limit` — items per request (default 25, max 100)
- `cursor` — opaque string encoding the position (base64-encoded last item key). Omit for first request.
- `next_cursor` — included in response when more results exist. `null` when at the end.
- `has_more` — boolean convenience field
- `sort` and `order` optional, sensible defaults per resource
- Filtering via query params: `?status=wanted`, `?monitored=true`, `?genre=28`

The cursor is opaque to the client — it encodes the sort key and ID of the last returned item. This gives stable pagination even as items are added or removed.

## Error responses

```json
{
  "error": {
    "code": "not_found",
    "message": "Movie with id 999 not found"
  }
}
```

HTTP status codes: 200, 201, 204, 400, 401, 404, 409, 422, 429, 500, 503.

- **429 Too Many Requests** — returned when an upstream provider (e.g.
  TMDB) rate-limits us and we haven't yet burned through retries. The
  response carries a `Retry-After: {seconds}` header mirroring the
  provider's guidance.
- **503 Service Unavailable** — returned when a required dependency
  (TMDB client, torrent session, Trakt, indexer aggregator) isn't reachable.
  Clients should retry with backoff rather than treat as fatal.

## Endpoints

The full endpoint catalogue is the OpenAPI spec, not this doc. The
in-binary spec (built from `utoipa` annotations on every handler) is
exported to:

- **`backend/openapi.json`** in the repo (regenerated by `just codegen`,
  drift-checked by `just codegen-check` on every CI run).
- **`https://docs.kinostack.app/reference/api`** — interactive viewer
  (Scalar) on the public docs site, refreshed on every release.
- The running binary itself serves it at `/api-docs/openapi.json` plus
  a Swagger UI at `/swagger-ui` for ad-hoc exploration.

A maintained markdown catalogue here would drift the moment a handler
ships; the OpenAPI spec is the contract. Auth model + pagination
envelope + error shape (above) are conventions every endpoint follows;
the per-route surface is in the spec.

## WebSocket

Real-time push for the UI. All events broadcast to all connected clients — no topic subscription needed for a single-user app. The client filters by what's relevant to the current view.

### Connection

```
ws://host/api/v1/ws
```

Authentication: send API key in the first message after connection:
```
Client → Server:
  {"type": "auth", "api_key": "xxx"}

Server → Client:
  {"type": "auth_ok"}
```

This avoids putting the API key in the URL (which leaks into logs and browser history).

### Protocol

```
Client → Server:
  {"type": "ping"}

Server → Client:
  {"type": "event", "topic": "movies", "action": "updated", "data": {...}}
  {"type": "event", "topic": "downloads", "action": "progress", "data": {"id": 5, "percent": 45.2, "speed": 5242880}}
  {"type": "event", "topic": "history", "action": "created", "data": {...}}
  {"type": "pong"}
```

### Event topics

| Topic | Actions | When |
|---|---|---|
| `movies` | created, updated, deleted | Movie added, status change, metadata refresh |
| `shows` | created, updated, deleted | Show followed, status change |
| `episodes` | created, updated | New episode detected, status change |
| `downloads` | created, updated, progress, deleted | Download started, state change, progress (throttled to 1/s), removed |
| `history` | created | Any new History event |
| `health` | updated | VPN, disk space, indexer status changes |
| `tasks` | started, completed, failed | Background task lifecycle |

The `data` field contains the full updated resource — clients don't need a follow-up GET.

Download `progress` events are throttled to 1 per second per download to avoid flooding the WebSocket.

## Expand parameter

For endpoints where related data is useful but not always needed:

```
GET /api/v1/movies/:id?expand=media,quality_profile
GET /api/v1/episodes/:id?expand=show,media
```

Available expansions per resource:
- **Movie**: `media`, `quality_profile`
- **Episode**: `show`, `series`, `media`
- **Media**: `streams`
- **Download**: `movie`, `episode`

Without `?expand`, only IDs are included for related resources. With expand, the full nested object is inlined.

## OpenAPI spec and client generation

The API is defined as an OpenAPI 3.x specification. This serves two purposes:

1. **Auto-generated TypeScript client** — hey-api generates a fully typed client from the spec. The Web UI imports this client directly. When the API changes, regenerate the client and the compiler catches any breakage. No manual fetch calls, no hand-written types.

2. **Documentation** — the spec is browsable via Swagger UI at `/api/docs` in development. Every endpoint, request body, response shape, and error code is documented in one place.

The OpenAPI spec is the source of truth for the API contract. The Rust server validates against it. The TypeScript client is generated from it. They can't drift apart.
